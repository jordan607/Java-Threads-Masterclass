🧵 What is a Thread?

A thread is a lightweight, independent path of execution within a process. Every Java program starts with a main thread. Multiple threads share the same heap memory but have their own stack, program counter, and local variables.

In the JVM, each thread maps to an OS-level thread (platform thread) — until Java 21's Virtual Threads changed that.

⚡ Why Threads?
  • Utilize multi-core CPUs fully
  • Non-blocking I/O — serve more requests
  • Background processing (async tasks)
  • Responsive UIs (keep UI thread free)
  • Parallelism — split heavy computation

Java Thread Model

🏗️ JVM Thread Architecture

Java threads are 1:1 mapped to OS threads (platform threads). The JVM's thread scheduler relies on the OS scheduler. Java 21 introduced Virtual Threads (Project Loom) with M:N mapping — millions of lightweight threads on few carrier threads.

ℹ️
Every Java program has at minimum: main thread, garbage collector threads, finalizer thread, and JVM internal threads. The JVM exits when all non-daemon threads complete.

Thread Lifecycle

A thread transitions through well-defined states from creation to termination. Understanding these states is critical for debugging and writing correct concurrent programs.

Thread States (java.lang.Thread.State)

NEW

Thread object created, start() not yet called. Thread not scheduled by OS yet.

RUNNABLE

Ready to run. May be actually running on CPU or waiting in the run queue for CPU time — JVM doesn't distinguish.

BLOCKED

Waiting to acquire a monitor lock (synchronized block held by another thread). Common source of contention.

WAITING

Indefinitely waiting — via Object.wait(), Thread.join(), or LockSupport.park(). Must be explicitly notified.

TIMED_WAITING

Waiting with timeout — Thread.sleep(ms), wait(timeout), join(timeout), parkNanos().

TERMINATED

Thread has finished execution (run() returned or threw uncaught exception). Cannot be restarted.

Checking Thread State in Code

ThreadStateDemo.java JAVA 
public class ThreadStateDemo {
    public static void main(String[] args) throws InterruptedException {

        Thread t = new Thread(() -> {
            try {
                Thread.sleep(2000); // TIMED_WAITING
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // restore interrupt flag
            }
        });

        System.out.println("Before start: " + t.getState());   // NEW
        t.start();
        Thread.sleep(100);
        System.out.println("During sleep: " + t.getState());  // TIMED_WAITING
        t.join();
        System.out.println("After join: "  + t.getState());   // TERMINATED
    }
}
⚠️
interrupt() contract: Always restore the interrupt flag when catching InterruptedException inside a Runnable/task. Use Thread.currentThread().interrupt() or rethrow. Never silently swallow it in production.
Chapter 03 — Fundamentals

Creating Threads

Java offers multiple ways to create threads. Understanding each approach — and when to use which — is foundational knowledge.

Method 1: Extend Thread

ExtendThread.javaJAVA
public class WorkerThread extends Thread {

    private final String taskName;

    public WorkerThread(String taskName) {
        super(taskName);                    // set thread name
        this.taskName = taskName;
    }

    @Override
    public void run() {
        System.out.printf("[%s] running on %s%n",
            taskName, Thread.currentThread().getName());
    }

    public static void main(String[] args) {
        WorkerThread t1 = new WorkerThread("Task-A");
        WorkerThread t2 = new WorkerThread("Task-B");
        t1.start(); // never call run() directly!
        t2.start();
    }
}

Method 2: Implement Runnable (Preferred)

RunnableExample.javaJAVA
public class DataProcessor implements Runnable {

    private final List<String> data;

    public DataProcessor(List<String> data) {
        this.data = data;
    }

    @Override
    public void run() {
        data.forEach(item ->
            System.out.println("Processing: " + item));
    }
}

// Usage
Runnable task = new DataProcessor(List.of("a", "b", "c"));
Thread t = new Thread(task, "data-thread-1");
t.start();

// Lambda (Java 8+) — most concise
Thread t2 = new Thread(() -> System.out.println("Lambda task"), "lambda-thread");
t2.start();

Method 3: Virtual Threads (Java 21+)

VirtualThreads.javaJAVA 21+
// Massive scale I/O-bound concurrency — millions of threads!

// Simple virtual thread
Thread vt = Thread.ofVirtual()
    .name("vt-1")
    .start(() -> System.out.println("Virtual thread!"));

// Virtual thread executor
try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 1_000_000).forEach(i ->
        exec.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1)); // non-blocking!
            return i;
        })
    );
} // auto-closes and awaits completion

// Virtual threads are cheap — no need for thread pools for I/O tasks
// Platform thread stack: ~1MB | Virtual thread stack: ~few KB (grows dynamically)
✅
Best Practice: Prefer Runnable over extending Thread — it separates the task from the execution mechanism and allows your class to extend other classes. For production, always use ExecutorService — never create raw threads manually in a server environment.
Chapter 04 — Synchronization

Synchronized & Locks

When multiple threads access shared mutable state, synchronization prevents race conditions and data corruption.

The Race Condition Problem

RaceCondition.javaBROKEN
public class Counter {
    private int count = 0;

    public void increment() { count++; } // NOT atomic! Read-Modify-Write
    public int  get()       { return count; }
}

// Two threads calling increment() 500 times each
// Expected: 1000 — Actual: 837, 912, 956... (random, data race)

Fix 1: synchronized keyword

SynchronizedCounter.javaJAVA
public class Counter {
    private int count = 0;

    // Method-level: lock is `this` object
    public synchronized void increment() { count++; }
    public synchronized int  get()       { return count; }

    // Block-level: finer granularity, better performance
    private final Object lock = new Object();
    public void safeIncrement() {
        synchronized(lock) {
            count++;
        }
    }

    // Static synchronized: class-level lock
    public static synchronized void staticMethod() { /* lock = Counter.class */ }
}

Fix 2: ReentrantLock (java.util.concurrent.locks)

ReentrantLockDemo.javaJAVA
import java.util.concurrent.locks.ReentrantLock;

public class BankAccount {
    private double balance;
    private final ReentrantLock lock = new ReentrantLock(true); // fair lock

    public void deposit(double amount) {
        lock.lock();
        try {
            balance += amount;
        } finally {
            lock.unlock(); // ALWAYS in finally!
        }
    }

    public boolean tryTransfer(double amount) {
        if (lock.tryLock()) { // non-blocking attempt
            try {
                if (balance >= amount) { balance -= amount; return true; }
            } finally { lock.unlock(); }
        }
        return false;
    }
}

// ReentrantLock advantages over synchronized:
// ✓ tryLock() — non-blocking, with timeout
// ✓ lockInterruptibly() — respond to interrupts
// ✓ Fair mode — prevents thread starvation  
// ✓ Multiple Condition objects (newCondition())
// ✓ Can span methods (lock in one, unlock in another — carefully)

ReadWriteLock — High-Throughput Reads

ReadWriteLockDemo.javaJAVA
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class CacheService {
    private final Map<String, String> cache = new HashMap<>();
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();

    public String get(String key) {
        rwLock.readLock().lock();      // multiple readers concurrent
        try { return cache.get(key); }
        finally { rwLock.readLock().unlock(); }
    }

    public void put(String key, String val) {
        rwLock.writeLock().lock();    // exclusive write
        try { cache.put(key, val); }
        finally { rwLock.writeLock().unlock(); }
    }
}

// Pattern: Cache with many reads, few writes → huge throughput gain
Chapter 05 — Synchronization

volatile & Atomics

For lightweight visibility and atomicity without full synchronization overhead.

volatile — Visibility Guarantee

VolatileFlag.javaJAVA
public class ServerTask implements Runnable {

    // Without volatile: CPU cache may hide the updated value from other threads
    private volatile boolean running = true;

    @Override
    public void run() {
        while (running) {
            processNextRequest(); // do work
        }
        System.out.println("Server stopped gracefully");
    }

    public void stop() {
        running = false; // guaranteed visible to run() thread immediately
    }
}

// volatile guarantees:
// 1. Visibility: writes immediately visible to all threads
// 2. Ordering: no reordering across volatile read/write (happens-before)
// 3. NOT atomic for compound operations: i++ is still 3 ops (read, modify, write)

AtomicInteger / AtomicLong / AtomicReference

AtomicsDemo.javaJAVA
import java.util.concurrent.atomic.*;

public class MetricsCollector {

    // Lock-free, CAS (Compare-And-Swap) based — very fast
    private final AtomicLong    requestCount  = new AtomicLong(0);
    private final AtomicInteger activeThreads = new AtomicInteger(0);
    private final AtomicBoolean isHealthy     = new AtomicBoolean(true);

    public void recordRequest() {
        requestCount.incrementAndGet();    // atomic increment
        activeThreads.incrementAndGet();
    }

    public void requestComplete() {
        activeThreads.decrementAndGet();
    }

    // CAS — optimistic locking pattern
    public boolean compareAndUpdate(int expected, int newValue) {
        return activeThreads.compareAndSet(expected, newValue);
    }

    // AtomicReference for object updates
    private final AtomicReference<Config> config = new AtomicReference<>();

    public void hotReloadConfig(Config newConfig) {
        config.set(newConfig); // atomic reference swap — no lock needed
    }
}

// LongAdder — even faster than AtomicLong under high contention
// Uses striping: each thread updates its own cell, sum = total
// Perfect for counters/histograms
private final LongAdder hitCounter = new LongAdder();
hitCounter.increment();
long total = hitCounter.sum();

Chapter 06 — Synchronization

wait / notify / notifyAll

Low-level thread communication via monitor objects. The foundation of the Producer-Consumer pattern.

⚠️
Rule: wait(), notify(), and notifyAll() must ALWAYS be called from within a synchronized block on the same monitor object, or you'll get IllegalMonitorStateException.

Producer-Consumer Pattern

ProducerConsumer.javaJAVA — Production Pattern
public class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) { this.capacity = capacity; }

    public synchronized void produce(T item) throws InterruptedException {
        while (queue.size() == capacity) {
            wait(); // releases lock, waits for consumer to drain
        }
        queue.add(item);
        notifyAll(); // wake waiting consumers
    }

    public synchronized T consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait(); // releases lock, waits for producer
        }
        T item = queue.poll();
        notifyAll(); // wake waiting producers
        return item;
    }
}

// CRITICAL: Always use while loop (not if) before wait()
// Reason: spurious wakeups — a thread can wake up without notify()
// Also: multiple consumers may have been notified, condition may be false again

// Modern alternative: BlockingQueue (prefer in production)
BlockingQueue<Task> queue = new LinkedBlockingQueue<>(100);
queue.put(task);    // blocks if full
Task t = queue.take(); // blocks if empty
notify() vs notifyAll()

notify() — wakes ONE random waiting thread. Use when all waiters are identical and you know only one slot opened.

notifyAll() — wakes ALL waiting threads (they re-compete for lock). Safer default. Slight performance cost.

Condition (modern alternative)

Use Lock + Condition for granular wait/signal. E.g., separate conditions for "not full" and "not empty" — avoids waking wrong threads.

Condition notFull  = lock.newCondition();
Condition notEmpty = lock.newCondition();
notFull.await();    // like wait()
notEmpty.signal(); // like notify()
Chapter 07 — Concurrency API

ExecutorService

In production, you never create raw threads. ExecutorService manages a pool of threads, reusing them for tasks — reducing overhead and providing lifecycle management.

🚫
Anti-pattern in production: new Thread(task).start() in every request. This creates unbounded threads, exhausts system resources, and has no error handling. Always use thread pools.

Thread Pool Types

ExecutorTypes.javaJAVA
import java.util.concurrent.*;

// 1. Fixed Thread Pool — bound CPU/memory, predictable
ExecutorService fixed = Executors.newFixedThreadPool(10);

// 2. Cached Thread Pool — grows/shrinks, good for short bursts (risky in prod)
ExecutorService cached = Executors.newCachedThreadPool();

// 3. Single Thread — serial execution, order preserved
ExecutorService single = Executors.newSingleThreadExecutor();

// 4. Scheduled — cron-like, recurring tasks
ScheduledExecutorService scheduled = Executors.newScheduledThreadPool(5);
scheduled.scheduleAtFixedRate(task, 0, 30, TimeUnit.SECONDS);

// 5. Work-Stealing Pool — ForkJoinPool under the hood (Java 8+)
ExecutorService workStealing = Executors.newWorkStealingPool();

// ⭐ PRODUCTION: Custom ThreadPoolExecutor — full control
ExecutorService executor = new ThreadPoolExecutor(
    10,                                    // corePoolSize
    50,                                    // maximumPoolSize
    60L, TimeUnit.SECONDS,                // keepAliveTime
    new LinkedBlockingQueue<>(200),       // bounded work queue
    new CustomThreadFactory("api-worker"),// named threads
    new ThreadPoolExecutor.CallerRunsPolicy() // backpressure
);

Lifecycle Management — Graceful Shutdown

GracefulShutdown.javaJAVA — Production Critical
@PreDestroy // Spring lifecycle hook
public void shutdown() {
    executor.shutdown(); // stop accepting new tasks
    try {
        if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
            executor.shutdownNow(); // interrupt running tasks
            if (!executor.awaitTermination(10, TimeUnit.SECONDS)) {
                log.error("Executor did not terminate cleanly");
            }
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}

// shutdown()     → graceful: drain queue, complete running tasks
// shutdownNow()  → forceful: interrupt all, return pending tasks

Chapter 08 — Concurrency API

Callable & Future

Runnable can't return values or throw checked exceptions. Callable solves both, and Future lets you retrieve the result asynchronously.

CallableFuture.javaJAVA
import java.util.concurrent.*;

// Callable — like Runnable but returns value and can throw
Callable<UserProfile> fetchUser = () -> {
    return userService.fetchFromDB(userId); // can throw Exception
};

ExecutorService exec = Executors.newFixedThreadPool(5);
Future<UserProfile> future = exec.submit(fetchUser);

// Do other work while user fetches...
doOtherWork();

try {
    UserProfile user = future.get(5, TimeUnit.SECONDS); // timeout!
} catch (TimeoutException e) {
    future.cancel(true); // interrupt the task
    throw new ServiceTimeoutException("User fetch timed out");
} catch (ExecutionException e) {
    throw e.getCause(); // unwrap the actual exception
}

// invokeAll — submit multiple, wait for all
List<Callable<Price>> priceTasks = List.of(
    () -> priceService.getStockPrice("TCS"),
    () -> priceService.getStockPrice("INFY"),
    () -> priceService.getStockPrice("WIPRO")
);
List<Future<Price>> results = exec.invokeAll(priceTasks, 10, TimeUnit.SECONDS);

// invokeAny — return result of FIRST to complete (race)
Price fastest = exec.invokeAny(priceTasks);
⚠️
future.get() blocks! Calling it without a timeout on an unbounded operation will hang your thread forever. Always use get(timeout, unit) in production. Also: future.isDone() for non-blocking checks.
Chapter 09 — Concurrency API

CompletableFuture

Java 8's CompletableFuture enables composable, non-blocking async pipelines — the backbone of reactive-style programming in standard Java.

CompletableFutureChain.javaJAVA 8+
import java.util.concurrent.*;

// Chain of async operations — no blocking
CompletableFuture
    .supplyAsync(() -> userRepo.findById(userId), executor)  // run async
    .thenApply(user -> enrichmentService.enrich(user))         // transform result
    .thenApplyAsync(user -> scoreService.score(user), executor) // async transform
    .thenAccept(result -> cache.put(userId, result))            // consume
    .exceptionally(ex -> { log.error("Failed", ex); return null; }); // error handle

// Combine two futures
CompletableFuture<UserProfile> userFuture  = fetchUserAsync(userId);
CompletableFuture<List<Order>> orderFuture = fetchOrdersAsync(userId);

CompletableFuture<Dashboard> dashboard = userFuture
    .thenCombine(orderFuture, (user, orders) ->
        Dashboard.build(user, orders)); // runs both in parallel, combines

// Wait for ALL to complete
CompletableFuture.allOf(userFuture, orderFuture)
    .thenRun(() -> System.out.println("All done!"));

// Wait for FIRST to complete (fastest-wins pattern)
CompletableFuture.anyOf(mirror1, mirror2, mirror3)
    .thenAccept(result -> useResult(result));

// Handle errors gracefully
CompletableFuture.supplyAsync(() -> riskyOp())
    .handle((result, ex) -> {
        if (ex != null) return fallback();
        return result;
    })
    .thenAccept(val -> respond(val));
Key Methods
  • supplyAsync — start async, returns value
  • runAsync — start async, no return
  • thenApply — transform (sync)
  • thenApplyAsync — transform (async)
  • thenCombine — merge two futures
  • thenCompose — flatMap (chain futures)
  • exceptionally — recover from error
  • handle — both success and error
  • whenComplete — side-effect, pass-through
⭐ Production Tip

Always pass an explicit Executor to supplyAsync / thenApplyAsync. Without it, they use ForkJoinPool.commonPool() which is shared across your entire JVM — a slow task can starve other operations.

// Always do this:
CompletableFuture.supplyAsync(
    () -> dbQuery(),
    ioExecutor  // your own pool
);

ConcurrentHashMapDemo.javaJAVA
ConcurrentHashMap<String, AtomicLong> hitCount = new ConcurrentHashMap<>();

// Thread-safe atomic operations
hitCount.computeIfAbsent(endpoint, k -> new AtomicLong()).incrementAndGet();

Chapter 11 — Production Patterns

Deadlock & Race Conditions

The two most dangerous concurrency bugs — both can bring down production systems silently.

Deadlock

💀
Deadlock: Thread A holds Lock 1, waits for Lock 2. Thread B holds Lock 2, waits for Lock 1. Both wait forever. CPU goes to 0%, system appears frozen.
DeadlockExample.javaBROKEN — DEADLOCK
public class DeadlockDemo {
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    // Thread 1: A → B
    static void thread1() {
        synchronized(LOCK_A) {
            Thread.sleep(100); // gives thread 2 time to grab LOCK_B
            synchronized(LOCK_B) { /* never reached */ }
        }
    }

    // Thread 2: B → A  ← opposite order = DEADLOCK
    static void thread2() {
        synchronized(LOCK_B) {
            synchronized(LOCK_A) { /* never reached */ }
        }
    }
}

// FIX: Always acquire locks in the SAME ORDER across all threads
// Or use: tryLock() with timeout → back off and retry
// Or use: lock ordering based on System.identityHashCode()
DeadlockPrevention.javaJAVA — Production Fix
// Fix 1: Lock ordering (canonical — always work with lower id first)
public void transfer(Account from, Account to, double amount) {
    Account first  = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to : from;
    synchronized(first) {
        synchronized(second) {
            from.balance -= amount;
            to.balance   += amount;
        }
    }
}

// Fix 2: tryLock with timeout — detect and back off
boolean gotBoth = false;
while (!gotBoth) {
    if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) {
        try {
            if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    doWork(); gotBoth = true;
                } finally { lockB.unlock(); }
            }
        } finally { lockA.unlock(); }
    }
    if (!gotBoth) Thread.sleep(ThreadLocalRandom.current().nextInt(10, 50));
}

When system hangs: kill -3 <pid> (JVM dumps to stdout) or jstack <pid>. Look for "Found one Java-level deadlock" in the output. Tools: VisualVM, JProfiler, Arthas, Async-profiler.

Chapter 12 — Production Patterns

Thread Pool Tuning

Choosing the right pool size is critical. Too few threads = underutilization. Too many = context switch overhead and OOM.

The Formula (Brian Goetz)

🧮 Optimal Thread Count
// CPU-bound tasks:
N_threads = N_cpu + 1  (extra for when a thread blocks on page fault etc.)

// I/O-bound tasks:
N_threads = N_cpu × (1 + Wait_time / Service_time)

// Example: 8 CPUs, task takes 20ms CPU + 80ms I/O
N_threads = 8 × (1 + 80/20) = 8 × 5 = 40 threads

// Get CPU count in Java:
int cpus = Runtime.getRuntime().availableProcessors();

Pool Configuration in Spring Boot

ThreadPoolConfig.javaSPRING BOOT — Production
@Configuration
public class ThreadPoolConfig {

    @Bean("ioExecutor")
    public ExecutorService ioExecutor() {
        int cpus = Runtime.getRuntime().availableProcessors();
        return new ThreadPoolExecutor(
            cpus * 4,                              // core: 4× CPU for I/O
            cpus * 8,                              // max: burst headroom
            60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(500),      // bounded!
            new ThreadFactoryBuilder()
                .setNameFormat("io-worker-%d")
                .setDaemon(true)
                .setUncaughtExceptionHandler(
                    (t, e) -> log.error("Uncaught in {}", t.getName(), e))
                .build(),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }

    @Bean("cpuExecutor")
    public ExecutorService cpuExecutor() {
        int cpus = Runtime.getRuntime().availableProcessors();
        return new ThreadPoolExecutor(
            cpus + 1, cpus + 1,               // fixed, no burst
            0L, TimeUnit.MILLISECONDS,
            new LinkedBlockingQueue<>(100),
            new ThreadFactoryBuilder()
                .setNameFormat("cpu-worker-%d").build()
        );
    }
}

Chapter 13 — Production Best Practices

Production Best Practices

Senior-level patterns used in real systems at scale — the things that distinguish production-grade concurrent code.

✅ The Production Checklist
  • Always name your threads — makes debugging thread dumps 10x easier
  • Set UncaughtExceptionHandler on all threads/pools — never lose exceptions silently
  • Always use bounded queues — unbounded queues lead to OOM under load
  • Never call Thread.stop() — deprecated and unsafe; use volatile flag or interrupt()
  • Don't use Thread.sleep() for coordination — use wait/notify or BlockingQueue
  • Prefer immutable objects for shared state — eliminates synchronization needs
  • Use ThreadLocal carefully — always clean up to avoid memory leaks in pools
  • Minimize synchronized scope — hold locks for the shortest time possible
  • Test with -ea (assertions) and stress test with more threads than CPUs
  • Monitor pool metrics in production — queue depth, active count, rejected tasks

ThreadLocal — Per-Thread Context

ThreadLocalDemo.javaJAVA — MDC / Request Context
// Classic use: per-request context propagation (user ID, trace ID)
public class RequestContext {
    private static final ThreadLocal<String> REQUEST_ID =
        ThreadLocal.withInitial(() -> UUID.randomUUID().toString());

    public static String getRequestId() { return REQUEST_ID.get(); }
    public static void   setRequestId(String id) { REQUEST_ID.set(id); }

    // CRITICAL: clean up or thread pool reuses thread with stale data!
    public static void clear() { REQUEST_ID.remove(); }
}

// Filter/Interceptor usage:
try {
    RequestContext.setRequestId(request.getHeader("X-Request-Id"));
    chain.doFilter(request, response);
} finally {
    RequestContext.clear(); // ALWAYS in finally
}

// Careful: doesn't work well with thread pools (inherited at creation time)

CountDownLatch & CyclicBarrier

Synchronizers.javaJAVA
// CountDownLatch — wait for N events (one-time gate)
CountDownLatch latch = new CountDownLatch(3); // wait for 3 services

startService("db",     () -> { initDB();    latch.countDown(); });
startService("cache",  () -> { initCache(); latch.countDown(); });
startService("kafka",  () -> { initKafka(); latch.countDown(); });

latch.await(30, TimeUnit.SECONDS); // main thread waits here
System.out.println("All services ready — accepting traffic");

// CyclicBarrier — rendezvous point, reusable
CyclicBarrier barrier = new CyclicBarrier(4, () ->
    System.out.println("All workers synced — next phase"));

// Each of 4 worker threads calls:
barrier.await(); // waits until all 4 arrive, then all proceed together

// Semaphore — limit concurrent access (rate limiter, connection pool)
Semaphore semaphore = new Semaphore(10); // max 10 concurrent DB connections
semaphore.acquire();
try { useDatabase(); }
finally { semaphore.release(); }
